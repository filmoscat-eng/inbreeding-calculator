## Что это и зачем

**Инбридинг** — это степень повышения гомозиготности, возникающая вследствие спаривания генетически родственных особей. Чем ближе родство родителей,
тем выше вероятность того, что у потомка две копии гена в локусе (одна от папы,
одна от мамы) окажутся **одинаковыми по происхождению** — то есть скопированы
с одного и того же гена общего предка.

Этот эффект численно выражается **коэффициентом инбридинга F**. Он:

- равен 0 у потомка двух неродственных особей;
- равен 0.25 у потомка полных сибсов;
- равен 0.5 теоретически у клонов (одна и та же особь скрещивается сама с собой,
  биологически это невозможно).

Высокий F — это плохо. Гомозиготные рецессивные аллели начинают проявляться,
жизнеспособность падает, репродуктивные показатели ухудшаются. В животноводстве,
ботанике, ветеринарии и сохранении видов F считают регулярно — он определяет,
можно ли скрещивать конкретную пару, и если можно, какие риски.

**Что делает эта программа:**

1. По загруженной родословной (CSV/XLSX) считает F **для каждой особи**.
2. Считает коэффициент родства **R(X, Y)** между любыми двумя особями.
3. Для конкретного пробанда показывает **разложение F по путям Райта** — через
   какого общего предка и каким именно «маршрутом» инбридинг накопился.
4. Прогнозирует F гипотетического потомка от любой пары особей.
5. Валидирует данные: ловит циклы в родословной, само-родительство,
   ссылки на несуществующих родителей.
6. Поддерживает родословные размером до десятков тысяч особей.

## Биологичесое обоснование: что такое F и R

### Identity by descent (IBD) — общее происхождение

В каждом локусе у диплоидного организма две аллели. Если обе пришли путём копирования
от **одного и того же гена общего предка** — это identity by descent (IBD).

`F(X) = вероятность IBD двух аллелей в случайно выбранном локусе X.`

В неинбредных популяциях F ≈ 0 (общий предок далеко в прошлом, разные мутации).
В замкнутой популяции с близкородственными скрещиваниями F быстро накапливается.

### Коэффициент родства R

R показывает, насколько две особи генетически похожи **по происхождению**.

| Отношение | R |
|---|---|
| Клон / монозиготные близнецы | 1 |
| Родитель ↔ потомок (неинбредный) | 0.5 |
| Полные сибсы | 0.5 |
| Полусибсы (один общий родитель) | 0.25 |
| Дядя ↔ племянник | 0.25 |
| Двоюродные | 0.125 |
| Двоюродные через одного | 0.0625 |
| Троюродные | 0.03125 |

R связан с f (коэффициентом коанцестрии) и F:

```
R(X, Y) = 2·f(X, Y) / sqrt((1+F(X)) · (1+F(Y)))
```

### Связь F и R через коанцестрию f

**Коэффициент коанцестрии f(X, Y)** — вероятность того, что **один случайно
выбранный ген X и один случайно выбранный ген Y** в одном локусе IBD.

Численно: `f(X, Y) = A[X, Y] / 2`, где A — аддитивная матрица родства.

Главное магическое равенство: **F будущего потомка от пары (X, Y) равен f(X, Y).**

Это и есть основа всех формул: считая f между двумя кандидатами на спаривание,
мы сразу получаем ожидаемый F их будущего потомка.

### Справочник стандартных типов спаривания

Если оба родителя пары неинбредные:

| Тип спаривания | F потомка |
| --- | --- |
| Неродственные | 0 |
| Двоюродные родственники | 0.0625 |
| Полусибсы (один общий родитель) | 0.125 |
| Полные сибсы (оба родителя общие) | 0.25 |
| Отец × дочь / Мать × сын | 0.25 |
| Дядя × племянница | 0.125 |
| Двойные двоюродные | 0.125 |

Программа после расчёта показывает ближайший по справочнику тип спаривания.

## Математика: откуда берутся формулы

### Метод Райта (1922) — формула с путями

Идея: F пробанда X равен вероятности IBD его двух аллелей в локусе. Аллели
пришли из разных родителей. IBD между папиной и маминой аллелями возможен,
если они обе унаследованы от **одного общего предка A** через два независимых
пути.

Если путь от папы S к предку A имеет длину n1 рёбер, и от мамы D к A — n2 рёбер,
то вероятность того, что:

- конкретная аллель попала из A в S через n1 шагов: (1/2)^n1
- конкретная аллель попала из A в D через n2 шагов: (1/2)^n2
- эти две аллели в A — одна и та же копия: 1/2 (А имеет 2 аллели, одна общая
  с вероятностью 1/2)
- A инбредный: добавляется (1 + F(A)) — потому что у инбредного предка две
  «как бы одинаковые» аллели увеличивают шанс совпадения.

Перемножаем и получаем вклад одного маршрута:

```
ΔF = (1/2)^(n1 + n2 + 1) · (1 + F(A))
```

И сумма по всем маршрутам через всех общих предков даёт F:

```
F(X) = Σ_A Σ_paths (1/2)^(n1 + n2 + 1) · (1 + F(A))
```

**Тонкости:**

- Перечисляются **простые пути** — без повторов вершин, иначе получится цикл.
- Перечисляются **пары непересекающихся путей** — пути от папы и мамы не
  должны иметь общих вершин кроме самого предка A. Иначе мы посчитаем один и тот же
  «материальный канал» наследования дважды.

### Табличный метод Хендерсона (1976) — рекуррентность

Райт удобен для понимания, но непрактичен для расчёта в больших родословных:
число путей растёт экспоненциально. Хендерсон предложил эквивалентный, но
вычислительно эффективный алгоритм.

Определим **аддитивную матрицу родства A** размера n×n, где n — число особей:

```
A[i, j] = 2 · f(i, j)              для i ≠ j
A[i, i] = 1 + F(i)
```

Если особь i имеет родителей s (отец) и d (мать), уже обработанных, то для
любой особи j (тоже уже обработанной) аллель j «двигается» к i через s или d:

```
A[i, j] = 0.5 · (A[s, j] + A[d, j])
```

Диагональ выводится из определения коанцестрии родителей:

```
A[i, i] = 1 + F(i) = 1 + f(s, d) = 1 + 0.5 · A[s, d]
```

**Это и есть весь алгоритм.** Перебираем особей в порядке поколений (родители
раньше потомков), заполняем матрицу строка за строкой. По диагонали получаем
F(i) = A[i, i] − 1.

Сложность: O(n²) по времени и памяти. На 1000 особей — миллион ячеек, это
ничто для современного компьютера.

### Численная эквивалентность Райта и Хендерсона

Метод Райта и метод Хендерсона дают **математически идентичный результат**.
В тестах проекта это явно проверено на нетривиальной родословной (см.
`tests/test_small.py::test_wright_paths_match_tabular`).

## Алгоритмы

### 1. Топологическая сортировка (алгоритм Кана)

**Зачем нужна:** табличный метод Хендерсона требует, чтобы для каждой особи i
все её j < i (в матрице) были уже обработаны. Это означает «родители раньше
потомков». Получаем такой порядок алгоритмом Кана:

1. Считаем для каждой особи **входящую степень** = число её родителей,
   присутствующих в родословной (0, 1 или 2).
2. В очередь кладём всех с нулевой степенью — это **основатели**.
3. Достаём из очереди, у всех её детей уменьшаем степень. У кого стала нулевой —
   в очередь.
4. Если по окончании остались особи с ненулевой степенью — это **цикл**.

Алгоритм заодно служит детектором циклов. Файл: `core/validators.py`.

### 2. Табличный метод (основной расчёт)

В коде (`core/algorithms.py::compute_relationship_matrix`):

```python
A = np.zeros((n, n))
for i, ind in enumerate(order):
    si = idx[father(ind)] or -1   # позиция отца
    di = idx[mother(ind)] or -1   # позиция матери
    if i > 0:
        if si >= 0 and di >= 0:
            row = 0.5 * (A[si, :i] + A[di, :i])
        elif si >= 0:
            row = 0.5 * A[si, :i]
        elif di >= 0:
            row = 0.5 * A[di, :i]
        else:
            row = np.zeros(i)
        A[i, :i] = row
        A[:i, i] = row
    A[i, i] = 1.0 + 0.5 * (A[si, di] if si >= 0 and di >= 0 else 0)
```

**Векторизация.** Внутренний цикл по столбцам j=0..i-1 заменён одним numpy-срезом
`A[i, :i] = ...`. Это убирает Python-цикл и даёт ~70× ускорение на 1000 особях.

### 3. Метод Райта (образовательный режим)

В коде (`core/algorithms.py::wright_paths`):

1. Найти всех **общих предков** отца и матери пробанда (пересечение
   множеств их предков).
2. Для каждого общего предка A:
   1. Перечислить все простые пути от отца до A через DFS.
   2. Перечислить все простые пути от матери до A.
   3. Для каждой пары (sire_path, dam_path), у которых нет общих вершин
      кроме A, добавить вклад `(0.5)^(n1+n2+1) · (1 + F(A))`.
3. Просуммировать.

**F(A) для общего предка** берётся из табличного метода (уже посчитан) — иначе
случай «инбредные общие предки» давал бы заниженный F.

**Ограничения.** Перечисление путей экспоненциально по сложности родословной.
В UI режим включается только для родословных ≤ 200 особей.

### 4. Расчёт R через матрицу A

```python
R(x, y) = A[x, y] / sqrt(A[x, x] * A[y, y])
```

(Так как `A[i, i] = 1 + F(i)`, формула эквивалентна классической Райта.)

### Что на странице

**Блок 1. Загрузка родословной.** Три способа: загрузить CSV/XLSX,
использовать встроенный пример (7 особей из исходного кода задачи),
сгенерировать синтетическую родословную ~1000 особей.

**Блок 2. Валидация.** Зелёная плашка либо подробное сообщение об ошибке.
Все ошибки описаны в разделе [Валидация](#валидация-и-сообщения-об-ошибках).

**Блок 3. F всех особей.**
Сортируемая таблица с колонками `id, father, mother, F, type`.
Карточки сверху: всего особей, инбредных (F > 0), максимальный F.
Кнопка «Скачать результат CSV».

**Блок 4. Анализ пробанда (метод Райта).**
Выпадающий список → выбираешь любую особь → видишь её F + классификацию.
Если F > 0 и родословная ≤ 200 особей — показывается **разложение по путям**:
таблица с колонками `общий предок, n1, n2, F(предка), вклад, путь отца, путь матери`.
Можно включить графовую визуализацию родословной до 8 поколений вверх.
Инбредные узлы подсвечены красным.

**Блок 5. Родство пары.** Два выпадающих списка → коэффициенты f и R + прогноз
F гипотетического потомка от этой пары.

## Программный API

```python
from core import (
    Pedigree,
    load_csv,
    load_excel,
    compute_inbreeding,
    compute_relationship_matrix,
    relationship,
    kinship,
    wright_paths,
    classify_inbreeding,
)

# === Загрузка из файла ===
ped = load_csv("examples/example_pedigree.csv")

# === F всех особей: dict[id -> F] ===
F = compute_inbreeding(ped)
print(F["E"])                      # 0.25

# === Коэффициент родства между двумя особями ===
print(relationship(ped, "C", "D")) # 0.5

# === Коанцестрия = ожидаемый F будущего потомка от пары ===
print(kinship(ped, "C", "D"))      # 0.25

# === Тип спаривания ===
print(classify_inbreeding(F["E"]).name)   # "Полные сибсы"

# === Разложение F пробанда по путям ===
total, contributions = wright_paths(ped, "G")
print(f"F(G) = {total:.4f} = сумма {len(contributions)} вкладов")
for c in contributions:
    print(f"  через {c.ancestor}: путь_отца={c.sire_path} (n1={c.n1}), "
          f"путь_матери={c.dam_path} (n2={c.n2}), вклад={c.contribution:.4f}")
```

### Ручное построение родословной без файла

```python
from core import Pedigree, compute_inbreeding

ped = Pedigree()
ped.add("A")                         # основатель (нет родителей)
ped.add("B")
ped.add("C", father="A", mother="B")
ped.add("D", father="A", mother="B")
ped.add("E", father="C", mother="D")
ped.validate()                       # явная валидация (опционально)

F = compute_inbreeding(ped)
print(F)
# {'A': 0.0, 'B': 0.0, 'C': 0.0, 'D': 0.0, 'E': 0.25}
```

### Получение полной матрицы родства

```python
import pandas as pd
from core import compute_relationship_matrix

A = compute_relationship_matrix(ped)         # numpy.ndarray (n, n)
print(A)

# Превратить в DataFrame для красивого вывода:
df_A = pd.DataFrame(A, index=ped.order, columns=ped.order)
print(df_A)
```

## Формат входных данных

### Минимальный CSV

```csv
id,father,mother
A,,
B,,
C,A,B
D,A,B
E,C,D
```

- **`id`** — строка, обязательное, уникальное в пределах файла.
- **`father`**, **`mother`** — пусто = основатель (родитель неизвестен или
  не учитывается).

### Алиасы колонок

Регистр и пробелы по краям не важны. Программа поймёт следующие варианты:

| Основные | Допустимые алиасы |
| --- | --- |
| `id` | `animal`, `name`, `individual`, `ид`, `имя`, `кличка` |
| `father` | `sire`, `papa`, `отец`, `папа`, `сир` |
| `mother` | `dam`, `mama`, `мать`, `мама`, `дам` |

То есть таблица с колонками `Кличка`, `Отец`, `Мать` загрузится.

### Что считается «пусто»

Пустая строка, `NaN`, `None`, `Null`, `—`, `-` — всё это нормализуется
до `None` (= родитель неизвестен).

### Лишние колонки

Колонки помимо id/father/mother **игнорируются**. Можно подсунуть экспорт из
родословной книги с десятком колонок (дата рождения, окрас, хозяин) — программа
возьмёт только три нужные.

## Полный разбор учебного примера

Возьмём родословную из файла `examples/example_pedigree.csv`:

```
A,B           — основатели
C = A × B     — потомок основателей
D = A × B     — полный сибс C
E = C × D     — потомок полных сибсов (инцест)
F = E × D     — потомок E и его собственной матери D
G = F × D     — потомок F и его бабушки D
```

### F(C) и F(D)

A и B — основатели, общих предков не имеют. F(C) = F(D) = 0.

### F(E)

Родители E — это C и D, у которых одни и те же родители A и B.

Общие предки C и D: {A, B}.

Через A:

- путь C→A: длина 1
- путь D→A: длина 1
- вклад: (0.5)^(1+1+1) · (1+0) = 1/8 = 0.125

Через B аналогично: 1/8 = 0.125.

**F(E) = 0.125 + 0.125 = 0.25.** Это «полные сибсы» по справочнику.

### F(F)

Родители F — это E и D. F(D) = 0. Но E сам инбредный (F(E)=0.25), и D — мать E.

Общие предки E и D: D сам (D — мать E, поэтому D присутствует во всех его предках),
плюс A и B.

Через D:

- путь E→D: длина 1 (E сразу имеет D как мать)
- путь D→D: длина 0 (это сама D)
- вклад: (0.5)^(1+0+1) · (1 + 0) = 1/4 = 0.25

Через A:

- путь E→C→A: длина 2
- путь D→A: длина 1
- вклад: (0.5)^(2+1+1) · (1+0) = 1/16

Через A через E→D→A: эти пути имеют общую вершину D (кроме A), значит не
независимы — пара отбрасывается.

Через B аналогично A: вклад 1/16.

**F(F) = 1/4 + 1/16 + 1/16 = 0.375.**

Программа выдаёт: `F = 0.3750`. Совпадает.

### F(G)

Родители G — F и D. Общие предки и пути ещё длиннее, программа находит 4 вклада:

```
через D: n1=1, n2=0, вклад = 0.2500
через D: n1=2, n2=0, вклад = 0.1250
через A: n1=3, n2=1, вклад = 0.0312
через B: n1=3, n2=1, вклад = 0.0312
```

**F(G) = 0.4375.**

Это родословная с очень высоким накоплением инбридинга — каждое следующее поколение
скрещивается с матерью предыдущего.

## Бенчмарки и масштабирование

```bash
.venv/bin/python benchmarks/bench.py
```

| Особей | Загрузка + валидация | Расчёт матрицы A | Пиковая память |
|---|---|---|---|
| ~110 | 6 мс | **1 мс** | 0.1 МБ |
| ~1 020 | 34 мс | **15 мс** | 8 МБ |
| ~5 040 | 168 мс | **292 мс** | 194 МБ |
| ~10 000 | расчётно ~700 мс | расчётно ~1.5 с | расчётно ~750 МБ |

**Цель «1000 особей < 1 секунды» перевыполнена в ~70 раз.**

### Почему так быстро

Векторизация. Внутренний цикл алгоритма Хендерсона по столбцам j=0..i-1 заменён
одним срезом NumPy. Это убирает overhead Python-интерпретатора (~50 нс на итерацию)
и даёт ускорение пропорционально длине строки.

### Когда O(n²) перестаёт работать

При n ≈ 50 000 матрица A содержит 2.5 млрд float64-ячеек = 20 ГБ памяти.
Это уже за пределами обычного компьютера. Для таких размеров нужен:

- **Алгоритм Meuwissen & Luo (1992)** — считает только F(i) без полной матрицы A,
  с O(n·m), где m — среднее число предков. Память O(n).
- **Разреженная инверсия A** через метод Quaas/Henderson — нужна для модели BLUP,
  но F даёт через диагональ.

Реализация любого из этих методов — задача на расширение проекта. Пока программа
оптимизирована для родословных до ~10 000 особей.

## Валидация и сообщения об ошибках

Все исключения наследуются от `core.PedigreeError`. UI ловит их и показывает
понятное сообщение пользователю. Для использования из кода:

```python
from core import (
    PedigreeError,           # базовый класс
    DuplicateIdError,        # дубли ID
    SelfParentError,         # X — свой собственный родитель
    UnknownParentError,      # ссылка на ID, которого нет в родословной
    SameParentError,         # отец == мать
    CycleError,              # цикл в родословной
)

try:
    ped = load_csv("data.csv")
except DuplicateIdError as e:
    print("Дубли:", e.ids)
except UnknownParentError as e:
    print(f"Особь {e.individual} ссылается на несуществующего {e.parent}")
except CycleError as e:
    print("Цикл:", " → ".join(e.cycle))
except PedigreeError as e:
    print("Прочая ошибка:", e)
```

### Что именно проверяется

1. **DuplicateIdError** — найдены повторяющиеся ID в исходных данных.
2. **SelfParentError** — особь указана как собственный отец или мать.
3. **UnknownParentError** — родитель указан, но его нет среди ID родословной.
4. **SameParentError** — отец и мать совпадают (биологически невозможно).
5. **CycleError** — обнаружен цикл (X → Y → ... → X). Невозможно в живой
   родословной, но возможен из-за опечаток в данных.

Сообщение об ошибке всегда содержит конкретные ID, чтобы пользователь
сразу нашёл проблемное место.

## Архитектурные решения

### Почему NumPy, а не Python-словари

Табличный метод Хендерсона — это плотные арифметические операции на матрице.
Python-словари дают O(1) доступ, но очень медленный для миллионов операций.
NumPy векторизует одну операцию на целую строку и работает на C-уровне.
Разница в ~70 раз на 1000 особях.

### Почему класс Pedigree, а не глобальный словарь

В исходном коде задачи был глобальный `pedigree = {}`. Это ОК для скрипта,
но плохо для веб-приложения и тестов:

- Несколько родословных в одной программе работать не будут.
- Тесты будут «загрязнять» друг друга.
- Кэш матрицы A некуда деть.

Класс инкапсулирует данные + кэш + валидацию.

### Почему два алгоритма (Хендерсон + Райт)

Хендерсон — для скорости и масштаба. Райт — для **объяснения**, через какого
предка и каким маршрутом инбридинг накопился. Это образовательная ценность,
которую не даёт матрица.

### Почему None для пустых родителей, а не пустая строка

`None` явно означает «значение отсутствует». Пустая строка — это «строка длины 0»,
семантически уже что-то. В классической родословной основатель — это особь без
известных родителей, и `None` это правильно отражает.

### Почему Streamlit, а не FastAPI + React

Streamlit — один Python-файл, никакого фронтенда писать не нужно, деплой в
один клик через Streamlit Community Cloud. Минус — UI кастомизируется ограниченно,
но для расчётной утилиты этого хватает.

### Почему отдельные модули core/

Разделение по слоям делает код тестируемым и переиспользуемым. UI можно поменять
(например, на FastAPI), не трогая ядро. Тесты пишутся к ядру, а не к UI.

## FAQ и типичные ошибки

### F получился отрицательный — это баг?

Нет, и да. Чисто математически F не может быть < 0. На практике из-за
плавающей точки очень маленькие отрицательные числа (~−1e-16) могут возникать
при численных шумах. Программа их обрабатывает: для классификации `F < 0`
автоматически приводится к 0. В сыром выводе `compute_inbreeding` они
возможны — фильтруйте `max(0, F)` если критично.

### Почему R(X, X) = 1?

R — это коэффициент родства. Для одной и той же особи он равен **диагональному
элементу матрицы A**, делённому на её же диагональ → всегда 1. Программа
действительно вернёт 1 для R(X, X). Но если особь сама инбредная, то A[X, X] > 1.

### Почему f(X, X) ≠ 0.5?

f(X, X) = 0.5 · (1 + F(X)), поэтому только у неинбредных особей f(X, X) = 0.5.
У инбредных — больше. Это покрыто тестом `test_kinship_self_equals_half_one_plus_F`.

### Почему добавление новой особи через ped.add() не пересчитывает F?

Любое изменение состава родословной сбрасывает кэш матрицы A. При следующем
вызове `compute_inbreeding` или `relationship` матрица будет пересчитана с нуля.
Это безопасный и предсказуемый дизайн.

### CSV не читается из-за кодировки

Передайте `encoding` в kwargs:

```python
ped = load_csv("данные.csv", encoding="cp1251")
```

`load_csv` пробрасывает все kwargs в `pandas.read_csv`.

### XLSX не читается

Убедитесь, что установлен `openpyxl`:

```bash
pip install openpyxl
```

Это опциональная зависимость в `requirements.txt`, но без неё pandas не читает .xlsx.

### Несколько листов в Excel-файле

```python
ped = load_excel("data.xlsx", sheet_name="Pedigree")
# или
ped = load_excel("data.xlsx", sheet_name=2)
```
